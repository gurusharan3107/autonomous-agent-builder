Autonomous Agent Builder Architecture Review — 2026-04-06

Question: Should we proceed with this architecture for a production-grade autonomous software builder at Accenture? (Python + FastAPI + SQLAlchemy + HTMX, polling orchestrator, 3 developer gates, Claude Code CLI, git worktrees, Semgrep/Trivy/Ruff quality gates, SQLite/PostgreSQL)

Council Verdict: Proceed — Conservative Path Only

Build the single-task execution loop end-to-end with full observability, hard failure modes, and working gates. Do not attempt the extensible platform or multi-agent scaling simultaneously. The three critical failures (polling, gates, SQLite) must be fixed before any release. Prove the core loop works first. You'll have better data for the ecosystem design after one successful deployment.

The One Thing to Do First: Build and validate a single end-to-end feature execution loop (user story → implementation → PR waiting for approval) with explicit gate rejection logic, full structured logging, and PostgreSQL. One sprint. Ship a demo showing a real feature reaching approval-waiting with Semgrep passing and a real PR created.

Where the Council Agrees

SQLite is a hard blocker for production Write-lock contention the moment you run parallel tasks. PostgreSQL from day one, not a future migration.
Polling orchestrator is a liability Latency jitter, operational opacity, masks timeout/recovery failures. Surface failure modes, don't hide them.
Developer gates are under-specified The rejection/remediation loop is undefined. What happens when a gate blocks work? Retry? Page a human? Fail the workflow? This is accountability theater without the spec.
Observability is non-negotiable for Accenture Agent broken code, gate blocking, orchestrator crashes — all invisible without structured logging and tracing. Day-one requirement.

Where the Council Clashes

Conservative path vs. Platform play First Principles + Executor: Ship one task end-to-end, prove gates work, then scale. Risk-reduction first.
Expansionist: Build pluggable agent protocol now for ecosystem moat. Multi-agent + webhooks create the real value.
Why they disagree: Conservative side says you can't validate ecosystem play without single-task data. Expansionist says waiting means missing the architectural window for extensibility.

Blind Spots the Council Caught

Agent code safety & blast radius No rollback, containment, or compute/network/storage limits per task. What stops a rogue agent from exfiltrating data or corrupting the repo?
Gate feedback loop is untested Gates block work but there's no spec for what happens next. Retry? Escalate? Fail? This loop must be falsifiable and testable.
Cost-per-decision transparency No token burn analysis or LLM overhead economics. Does a planning task cost $0.50 or $5.00? Accenture is cost-conscious.
Claude Code CLI is a single point of failure Network timeouts, recovery, deadlocks — all unspecified. The entire orchestrator depends on CLI calls succeeding.
Multi-agent merge conflicts No conflict resolution strategy when parallel agents modify the same file. Git worktrees don't solve this alone.

Advisor Responses

The Contrarian

Three critical failure points:

  1. Polling orchestrator is wrong. Polling with exponential backoff means latency jitter. Enterprise clients demand predictable SLAs. You're copying Symphony's design without its justification. Need event-driven dispatch with a task queue (Celery, RQ, or Redis pubsub). Polling also constantly hits the database; at scale this becomes a bottleneck.
  2. The 3 developer gates punt on the hardest problem. Automating 95% of SDLC but keeping humans for the most subjective parts. The gap between "looks good to me" and "works in production" is where enterprise failures happen. Need objective gates. Push humans to override gates, not approve designs.
  3. SQLite for multi-agent concurrency is a disaster. Two parallel agents = write-lock contention. PostgreSQL is non-negotiable from day one.

Bottom line: Cut the polling. Harden the gate framework. Drop SQLite. Then you have something real.

The First Principles Thinker

You're solving the wrong problem. Building an orchestration framework when you need a task execution model. The architecture conflates two distinct concerns: how to coordinate delivery (business) with how to safely run untrusted code (technical).

The 3 developer gates don't distribute evenly. 80% autonomous work between gates means gates become accountability theater unless you've deeply thought through what information humans need to make those decisions.

What's actually hard: Defining what "planning gate approved" means. What does quality gate remediation do when Semgrep finds 47 violations in autogenerated code?

What's not hard: Running agents in git worktrees and collecting results. That's mechanics.

Recommendation: Before building orchestration, ship a single-task execution loop end-to-end with full observability. Prove the gates work. Then generalize. Right now you're scaling uncertainty.

The Expansionist

Proceed and expand the scope. This architecture is undershooting the opportunity.

  • Multi-agent collaboration: Git worktrees + isolated agents = foundation for parallel agents with dependency resolution and merge conflict resolution.
  • Continuous learning: Quality gates + remediation success/failure = training data. After 6 months, gates become predictive, not reactive. Defensible moat.
  • Agent infrastructure play: Make the protocol so clean customers can plug in their own agents. Suddenly this isn't a builder tool; it's enterprise DevOps infrastructure.
  • Platform extensibility: Add webhooks + event streaming so clients build their own dashboards on top.

Demo implication: Don't demo "we built a feature." Demo "we built something your teams can extend."

The Outsider

The architecture is sound, but you're shipping a complexity tax that Accenture clients won't understand.

  1. Polling will mystify stakeholders. "Why isn't the task starting?" The answer (poll queue with exponential backoff) sounds like a workaround.
  2. Developer gates = babysitting. The moment a planning review gets stuck for 2 hours, clients blame the tool. Make criteria explicit and testable before hitting human gates. Auto-reject tasks that don't meet the bar.
  3. SQLite for enterprise is a landmine. PostgreSQL is table stakes for "production-grade."

The rest is solid. HTMX over React is refreshingly sane. Semgrep + Trivy + Ruff is industry standard. Claude Code as agent runner is defensible.

The Executor

Yes proceed. This is doable and the path is clear. FastAPI + SQLAlchemy is a solved problem. HTMX cuts complexity. Git worktrees exactly right.

Prove these in 1-day spikes:

  1. Orchestrator polling loop works without deadlock
  2. Claude Code CLI integration captures structured results and handles failures
  3. Quality gates are <100 lines each with working Semgrep auto-remediation
  4. Approval gates have a working walkthrough (approve/reject flow)

What will kill you:

  • Trying Java/Node/Python in MVP — pick ONE, prove it, add others later
  • Over-speccing the dashboard — kanban + task detail + approval screen, that's it
  • Agent callback complexity — task starts, completes, gate passes/fails. No event streaming yet.

Green light: 2-minute video of feature going from description to approval-waiting with real Semgrep results and a real PR.
