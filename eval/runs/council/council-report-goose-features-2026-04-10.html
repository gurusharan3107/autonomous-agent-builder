Goose Feature Incorporation Plan — autonomous-agent-builder v4.1 — 2026-04-10

Question:

Should we incorporate 6 Goose-inspired features (Subagent Parallel Dispatch, Context Compaction, Recipe System, Prompt Injection Detection, Egress Monitor, Permission Store) into our deterministic, safety-first SDLC orchestrator built on Claude Agent SDK? Is the prioritization correct? Are there architectural risks?

Chairman's Verdict

The Recommendation

Do not adopt the 6 Goose features as a package. Instead:

  • Immediately adopt (P0): Prompt Injection Detection, Egress Monitor, Permission Store — security/audit, directly sells to enterprise, doesn't touch orchestration.
  • Measure then decide (conditional P0): Instrument phase telemetry for 2 weeks. If context exceeds 70% causing failures → Context Compaction (as explicit phase budgets + token caching, not lossy compression). If sequential phases bottleneck → Parallel Dispatch (at orchestrator level, not agent level).
  • Decouple Recipe System: Define what "recipe" means for your model first. Likely P0.5 (operational tax).
  • Reject as-is, redesign if needed: Subagent Dispatch only as orchestrator-level DAG parallelism. Context Compaction only as phase budgets + token caching.

Determinism is non-negotiable for enterprise. Any feature that makes replay or auditability harder doesn't go in.

The One Thing to Do First

Instrument your current system with phase-level telemetry (duration, token usage, tool_registry call count, context % at phase start/end, recovery latency on failure). Run it against production workloads for 2 weeks. Share the data. Then prioritize based on actual constraints, not feature envy.

Where the Council Agrees

  • Context Compaction via lossy summarization is structurally risky. Every advisor who engaged with it flagged that lossy compression in a deterministic, safety-critical pipeline violates the enterprise contract. (Contrarian, First Principles, Executor all converged.)
  • Subagent Parallel Dispatch contradicts deterministic orchestration as currently designed. Partial-state recovery, ordering nondeterminism, and tool_registry mutation all remain unaddressed.
  • Goose features were adopted as a feature list, not derived from constraints. No evidence that Parallel Dispatch or Context Compaction are blocking production workloads. "What's broken now?" was never answered.
  • Security/audit features are defensible regardless. Injection Detection, Egress Monitor, Permission Store address enterprise compliance directly and don't interfere with determinism.

Where the Council Clashes

Is parallelism a natural evolution or a philosophical break?

  • First Principles + Contrarian: Parallelism is a fundamental break. Once agents run in parallel, recovery, auditability, and replay all become harder. The tool_registry contract assumes sequential discovery.
  • Executor + Outsider: Parallelism within a phase, under orchestrator control, is fine. The question is whether you're parallelizing agent dispatch (breaks determinism) or task execution (doesn't, if tasks are independent).

Is Context Compaction a blocker or premature?

  • Executor: If you hit 80% context mid-phase, you fail. This needs to land before production.
  • Contrarian + First Principles: If phases are designed correctly with explicit budgets, you never hit 80%. Compaction is a symptom of bad phase design.

Blind Spots the Council Caught

  • Tool_registry mutation: If Parallel Dispatch, Recipe System, or Permission Store cause tools to appear/disappear mid-phase, session resumption via resume=session_id breaks irreparably. This wasn't in the original plan. (Caught by Review 2)
  • Determinism was never defined: Does it mean reproducible routing? Reproducible agent outputs? Reproducible tool availability? The answer changes the verdict on Parallel Dispatch. (Caught by Review 5)
  • No measurement baseline exists: Can't prioritize P0 vs P1 without phase latency, context usage per phase, and failure frequency data. (Caught by Reviews 4 & 5)
  • Recipe System purpose is ambiguous: Is it versioned prompt templates? Customer workflow scaffolding? Agent specialization? The answer determines if it's table stakes or gold-plating. (Caught by Review 3)

Advisor Responses

Contrarian — "Start over with: what's broken now?"

Subagent parallel dispatch is fundamentally at odds with your deterministic orchestration claim. You've traded determinism for parallelism without explaining how you handle partial state. If agent-2 hits a token limit mid-implementation while agent-1 succeeds, what's your recovery model?

Context compaction at 80% is a hidden landmine — lossy compression in a safety-critical pipeline is unacceptable for enterprise.

Recipe system + permission store assume tool decisions are reusable, but they're not in adversarial contexts. Enterprise security teams will rightfully ask: if a permission is cached, how do you detect when threat models shift?

The real gap: You haven't addressed WHY Goose features fit your constraints. Start over with: "What's broken now that these features fix?" Not "Goose has this, so we should too."

First Principles — "You're asking the wrong question"

The Goose features are agent-centric; your system is deterministic. Three features will compromise determinism; two exploit non-existent infrastructure; one is security theater.

Subagent dispatch breaks determinism (moves control flow into agents). Context compaction breaks session resumption via lossy summarization. Recipe system parameterizes your agent-as-artifact model (doctrine drift). Prompt injection regex won't catch sophisticated attacks.

Alternatives: Parallelize at the orchestrator level, not the agent level. Use token caching + explicit phase budgets instead of lossy compaction. Double down on workspace isolation for security.

Expansionist — "This is undershooting the opportunity"

The 6 Goose features are table stakes — solid engineering tax — but they don't unlock the real value hiding in your architecture.

What you should be chasing: (1) Multi-agent task decomposition at the DAG level. (2) Agent specialization marketplace — swap agents at runtime based on task complexity/domain. (3) Cost-aware task routing with learned agent-to-task mapping. (4) Audit trail as a compliance product.

Turn this from an SDLC builder into an enterprise AI task platform that learns.

Outsider — "Flip your prioritization entirely"

No evidence Subagent Parallel Dispatch is blocking anything today — measure first. Context Compaction same issue — might not exist at production scale.

Recipe System I get for enterprise, but should emerge from actual customer workflows, not Goose.

Flip priorities: Do Injection Detection, Egress Monitor, Permission Store first (P0). They're security/audit and directly sell to enterprises. Then measure whether Parallel Dispatch and Compaction actually matter.

Executor — "Prioritization is wrong"

Subagent Parallel Dispatch (current P0) is a scaling optimization that matters only after you've proven the sequential baseline works at enterprise scale.

Context Compaction (current P0) is a blocker disguised as optimization — if you hit 80% context mid-phase, you either fail the task or compact. There's no option C.

Recipe System (current P1) is actually P0.5 — it's operational scaffolding. Without it, every new customer variant becomes a code branch.

First step Monday: Measure baseline phase latency. Then implement Context Compaction + Recipe System.
