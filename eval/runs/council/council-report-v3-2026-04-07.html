LLM Council Report — v3

Autonomous Agent Builder Architecture · 2026-04-07 · Round 2 (post-research)

Question: Is the v3 architecture (concurrent gates, Langfuse, group chat approval, CAPABILITY_LIMIT, harnessability scorer, deterministic dispatch) ready to implement?
Context: v2 council fixed SQLite, polling, gate spec, observability, agent safety, CLI resilience. v3 adds research-based improvements.

Verdict: Ship v3 — with 5 blockers resolved before first line of code

The architecture is sound and significantly improved from v2. The concurrent gate design, Langfuse integration, and deterministic dispatch are all correct choices. However, 5 under-specifications will cause production failures if not resolved now: gate-level observability, harnessability rubric, CAPABILITY_LIMIT flow, approval audit trail, and group chat authority model.

The One Thing to Do First: Write HARNESSABILITY.md with a concrete rubric — 3–4 scoring rows with examples of what score 5, 6, 7, 8 mean. Score 3 real projects manually. If it takes >30 min/project, simplify. Everything else (routing rules, architect review gate, enforcement) depends on this.

Where the Council Agrees

Concurrent gates need per-gate timeouts + circuit breakersIf Semgrep hangs >30s, kill it and emit TIMEOUT (not FAIL). Prevents cascading hangs that choke the entire pipeline.
Group chat approval is not RBAC — needs authority modelWho approves? Can a developer approve their own PR? What's the escalation if the approver is absent? Needs explicit authority rules before UI is built.
Harnessability scorer needs enforcement rules, not just a scoreScore <5 → architect review required. Score <3 → reject task. Without routing logic, the score is decoration.
Langfuse is under-leveraged for gate-level observabilityToken costs are tracked, but which gate failed, which timed out, and why is not specified. Gate-level traces are required for debugging concurrent AND aggregation.

Where the Council Clashes

Nightly arch gate vs. PR heuristicFirst Principles: 24h lag kills feedback loops — run fast heuristic on PR.
Executor: Nightly is right for Phase 1 scope — add heuristic in Phase 2.
Resolution: Nightly ships Phase 1 (pragmatic). Design it for 5-min runtime and document the Phase 2 PR heuristic migration path now.
Scope of Phase 1 (Expansionist vs. Executor)Expansionist: Polyglot dispatch and Semgrep-as-training-data are Phase 1 architectural decisions.
Executor: Phase 1 = one task, one language, ship fast.
Resolution: Executor wins. Document Expansionist's observations as design artifacts for Phase 2, not v3 code.

Blind Spots the Council Caught

AND aggregation observability is darkWhen gate B times out and gate C errors, the aggregate shows "FAIL" but not which gate failed or why. Every gate needs a structured Langfuse event with gate name, status type (timeout vs. fail vs. error), elapsed_ms, and error_code.
CAPABILITY_LIMIT has no queue strategyTask enters CAPABILITY_LIMIT — then what? Sits indefinitely? Gets rejected? Goes to a dead-letter queue? Define: rejected tasks enter a dead-letter queue, analyst has 4h SLA to review before feature is rejected.
Per-gate timeout is missing from concurrent designasyncio.gather does not time out individual gates. Wrap each gate call in asyncio.wait_for(gate.run(), timeout=GATE_TIMEOUTS[gate_name]) with specific timeouts: Ruff=30s, pytest=120s, Semgrep=60s, Trivy=45s.
Token-per-task ceiling missingCAPABILITY_LIMIT prevents infinite retries, but a single run can still burn unbounded tokens. Add MAX_TOKENS_PER_TASK to config. When exceeded, mark task BLOCKED and notify developer before the next agent call.

Advisor Responses

The Contrarian

4 Fatal Gaps:

  1. Concurrent gates + AND logic = catastrophic brittleness. No per-gate timeouts, no circuit breaker when Langfuse is unreachable. A flaky external service kills the enterprise demo.
  2. Group chat approval is underspecified theater. No RBAC, no audit trail, social consensus ≠ enterprise-grade. Accenture clients need role-based access and immutable approval records.
  3. CAPABILITY_LIMIT escalation path is broken. Escalate to what queue? What SLA? Adds a new failure mode (escalation backlog) without defining resolution.
  4. Harnessability scorer has no teeth. Score 3/8 → what happens? No enforcement rules means it's a report nobody acts on.
The First Principles Thinker

Not ready. Three issues:

  1. Concurrent gates is load-shifting, not risk-reduction. You're still as strong as your weakest gate (likely Semgrep). AND aggregation is correct, but don't let concurrency create false confidence.
  2. Harnessability scorer needs routing rules: score<5 → architect reviews; <3 → reject. Without rules, it's documentation theater.
  3. Nightly architecture gate creates a 24h feedback lag. Run fast heuristic on PR, defer deep analysis to nightly.
The Expansionist

Enterprise-solid but too narrow. Three plays:

  1. Semgrep captures a diff on every autofix run. That diff is training data for AGENTS.md tuning — what patterns keep failing?
  2. Group chat approval transcripts = preference data for a critic model. One schema change away from fine-tuning your own reviewer.
  3. Polyglot gate dispatch should be a Phase 1 architecture decision (abstract over linter/fixer). Multi-language is not a Phase 2 feature — it's a Phase 1 unlock.
The Outsider

Four underspecifications:

  1. AND aggregation precedence unclear when one gate times out vs. fails — state machine needed.
  2. Approval decision must be committed to DB — not just logged to Langfuse. Langfuse is observability; DB is audit trail.
  3. Nightly gate failure post-merge = rollback trigger or next-day fix? Policy must be defined.
  4. Token-per-task ceiling is missing. CAPABILITY_LIMIT prevents retries but not unbounded single-run burn.
The Executor

Ship it. The path is clear and manageable.

Concurrent gates = testable, observable. Semgrep two-step mirrors human review. Langfuse + LiteLLM = cost predictable. CAPABILITY_LIMIT = clear circuit breaker. Deterministic dispatch = kills distributed systems headache.

5 things before coding:

  1. Prototype group chat UI — it's the UX binding. Decide authority model first.
  2. Nail harnessability rubric — write it, score 3 projects, simplify if >30 min each.
  3. Define nightly gate SLA — what triggers it, what happens on 2am failure, does it block next-day deploys?
  4. Write CAPABILITY_LIMIT flow doc — one page diagram.
  5. Draw approval UI wireframe — what does the developer actually click?
