# LLM Council Transcript — 2026-04-06

## Original Question

Should we proceed with this architecture for a production-grade autonomous software builder at Accenture?

## Framed Question

**Decision:** Should we proceed with this architecture for a production-grade autonomous software builder at Accenture?

**Context:** Building a system that autonomously takes a feature description through the full agile SDLC — planning, design, implementation, quality gates, PR creation, build verification. Inspired by OpenAI's Symphony (orchestrator + isolated workspaces), but fully open-source. For Accenture enterprise clients.

**Key design choices to evaluate:**
1. Python + FastAPI + SQLAlchemy + SQLite/PostgreSQL stack
2. HTMX dashboard (no React SPA) for the built-in board/kanban
3. 3 developer gates (planning, design, PR review) — everything else autonomous
4. Quality gates: Semgrep, Trivy, Ruff/ESLint/Checkstyle with auto-remediation
5. Claude Code CLI as primary agent runner + LiteLLM for pluggability
6. Git worktrees for workspace isolation per task
7. Polling-based orchestrator (like Symphony) with exponential backoff
8. 4-phase implementation: MVP → Quality Gates → Security + Full SDLC → Enterprise

**Stakes:** This will be demoed to Accenture enterprise clients. Must be production-grade, not a prototype. Must handle Java/Node/Python. No proprietary SaaS dependencies.

## Advisor Responses

### The Contrarian

Three critical failure points:

1. **Polling orchestrator is wrong.** Polling with exponential backoff means latency jitter — sometimes tasks sit idle for minutes. Accenture enterprise clients will demand predictable SLAs. You're copying Symphony's design without its justification (isolated, untrusted workspaces). Your workspaces are still within the same repo boundary. You should be using event-driven dispatch with a task queue (Celery, RQ, or even simple Redis pubsub). Polling also means you're constantly hitting the database; at scale this becomes a bottleneck and cost sink.

2. **The "3 developer gates" design punts on the hardest problem.** You're automating 95% of SDLC but keeping humans in the loop for the most subjective parts (planning validation, design review, PR approval). The gap between "this looks good to me" and "this will actually work in production" is where enterprise failures happen. You need objective gates — not humans gatekeeping subjective questions. Automate unit coverage thresholds, architecture rules, security posture checks. Push humans to override gates, not approve designs.

3. **SQLite for multi-agent concurrency is a disaster waiting to happen.** The moment you have two agents running parallel tasks, you'll hit write-lock contention. PostgreSQL is non-negotiable here.

Cut the polling. Harden the gate framework. Drop SQLite. Then you have something real.

### The First Principles Thinker

You're solving the wrong problem. You're building an orchestration framework when what you actually need is a task execution model. The architecture conflates two distinct concerns: how to coordinate software delivery (the business problem) with how to safely run untrusted code (the technical problem).

Strip away the Accenture demo context. What's the core constraint? You need to run Claude Code agents in isolation, with observable gates, and guarantee that bad outputs don't corrupt the repository. That's it.

Your current design treats the orchestrator as the bottleneck: it polls, dispatches, waits, retries. But polling is a symptom of assuming you need to coordinate tasks. You don't. You need to commit task boundaries.

**Actually hard:** Defining what "planning gate approved" means. When does a design document satisfy a human reviewer? What does "quality gate remediation" do when Semgrep finds 47 violations in autogenerated code?

**Not hard:** Running agents in git worktrees and collecting results. That's mechanics.

**The deeper issue:** Your 3 developer gates don't distribute evenly across a full SDLC. You've got 80% of the work happening autonomously between gates, which means the gates become accountability theater unless you've deeply thought through what information humans need to make those decisions.

**Recommendation:** Before building orchestration, ship a single-task execution loop end-to-end (feature → implementation → PR) with full observability. Prove the gates work. Then generalize. Right now you're scaling uncertainty.

### The Expansionist

Yes, proceed — and expand the scope. This architecture is undershooting the actual opportunity. You're building a task orchestrator when you could be building a capability platform.

1. **Git worktrees + isolated agents** is the foundation for multi-agent collaboration. Run parallel agents on different features, with dependency resolution, merge conflict resolution, and cross-feature impact analysis. Trivial to add once workspace isolation is there.

2. **Quality gates + auto-remediation** is gold for continuous learning. Every gate success/failure = training data. After 6 months, your quality gates become predictive, not reactive. Defensible moat.

3. **Claude Code CLI as agent runner** — LiteLLM gives escape velocity. But expand: make the agent protocol so clean that customers can plug in their own agents. Suddenly this isn't a builder tool; it's an agent infrastructure play for enterprise DevOps.

4. **HTMX dashboard** keeps it lightweight. Add webhooks + event streaming so clients can build their own dashboards on top. You're not a UI company; you're a platform.

**Demo implication:** Don't demo "we built a feature." Demo "we built something your teams can extend."

### The Outsider

The architecture is sound, but you're shipping a complexity tax that Accenture clients won't understand.

1. **Polling orchestrator will mystify stakeholders.** "Why isn't the task starting?" sounds like a workaround, not a design choice. Consider a task dispatch queue instead.

2. **Three developer gates sounds right until it doesn't.** You've created a checkpoint system where humans must babysit an "autonomous" system. The moment a planning review gets stuck for 2 hours, clients blame the tool. Make criteria explicit and testable before hitting the human gate. Auto-reject tasks that don't meet the bar. Or accept this is a human-in-the-loop system.

3. **SQLite for enterprise is a landmine.** PostgreSQL is table stakes. SQLite is a visible gap that undermines trust.

**The rest is solid.** HTMX over React is refreshingly sane. Semgrep + Trivy + Ruff stack is industry standard.

### The Executor

Yes, proceed. This is doable and the path is clear. FastAPI + SQLAlchemy is a solved problem. HTMX over React cuts complexity. Git worktrees for isolation is exactly right.

**Can you ship the MVP in 4-6 weeks?** Prove it:

1. Orchestrator polling loop works — not theory, working prototype that dispatches and doesn't deadlock. One day.
2. Claude Code CLI integration is real — invoke programmatically, capture structured results, handle failures. One day.
3. Quality gate framework — each gate <100 lines. Semgrep + one auto-remediation end-to-end. One day.
4. 3 approval gates walkthrough — working approve/reject flow.

**What will kill you:**
- Trying Java/Node/Python in MVP. Pick one. Prove it. Add others in phase 2.
- Over-speccing dashboard. Kanban + task detail + approval screen. That's phase 1.
- Agent callback complexity. Task starts, completes, gate passes/fails. No event streaming yet.

**Green light:** 2-minute video of feature going from description → approval-waiting with real Semgrep results and a real PR.

## Chairman's Verdict

### Where the Council Agrees

1. **SQLite is a hard blocker for production.** PostgreSQL from day one, not a refinement.
2. **Polling orchestrator is a liability.** Latency jitter, operational opacity, masks failures.
3. **The three developer gates are under-specified.** Rejection/remediation loop undefined.
4. **Observability is non-negotiable.** Day-one requirement for Accenture.

### Where the Council Clashes

**Conservative path vs. Platform play.** First Principles + Executor say prove single-task loop first. Expansionist says build pluggable agent protocol now. The resolution: you cannot validate an ecosystem play without observability data from single-task execution. Attempting both simultaneously increases shipping time and risk without increasing confidence.

### Blind Spots the Council Caught

1. Agent code safety & blast radius (rollback, containment, resource limits)
2. Gate feedback loop untested (what does rejection trigger?)
3. Cost-per-decision transparency (token burn economics)
4. Claude Code CLI single point of failure (timeout/recovery unspecified)
5. Multi-agent merge conflicts (no conflict resolution strategy)

### The Recommendation

Proceed on the conservative path. Build the single-task execution loop end-to-end with full observability, hard failure modes, and working gates. Do not attempt extensible platform or multi-agent scaling simultaneously.

### The One Thing to Do First

Build and validate a single end-to-end feature execution loop (user story → implementation → PR waiting for approval) with explicit gate rejection logic, full structured logging, and PostgreSQL. One sprint.
