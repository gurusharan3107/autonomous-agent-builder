Per-Project Structured Memory System — 2026-04-14 — 5 advisors, 5 peer reviews

Chairman's Verdict

Do not ship as-is. The fundamentals are sound — pull-based retrieval, frontmatter discipline, and separating concerns are all correct. But three architectural gaps need design fixes before implementation:

1. Replace 1h TTL cache with write-through invalidation on routing.json

2. Scrap keyword-overlap contradiction detection — use dead-letter queue for human review

3. Remove graduation rule or make it explicit (user/agent decides, not grep)

Then specify: consumption patterns, concurrency model, cold-start strategy, and correction SLA.

The One Thing to Do First: Build a test harness with 10 realistic memory files and 12 queries. Wire retrieval end-to-end with logging. This surfaces consumption pattern gaps and scoring assumptions before touching GC or caching.

Where the Council Agrees

  • Pull-based retrieval is architecturally sound. All advisors converged. Always-loaded memory is a token sink; progressive disclosure is correct.
  • Routing cache 1h TTL is broken. Contrarian + First Principles both identified the write-read consistency bug independently.
  • Contradiction detection via keyword overlap is unreliable. False positives guaranteed at scale. Not a tuning problem — fundamentally weak signal.
  • Graduation at 60% grep is pattern-matching theatre. Three advisors rejected this independently.
  • System lacks operational grounding. No consumption patterns, usage budgets, concurrent write handling, or adoption strategy specified.

Where the Council Clashes

  • Ship now vs. fix first: Expansionist says frontmatter + retrieval is strong enough to ship and iterate. All others say the cache, contradiction, and graduation bugs need design fixes, not operational workarounds.
  • Mechanical GC vs. configurable policy: Executor wants GC rules configurable per-project. Contrarian + Expansionist say the 4 rules are defensible defaults.
  • Who bears contradiction risk: Expansionist accepts human review as control. Outsider + Executor want the system designed to not require that oversight level.

Blind Spots Caught in Peer Review

  • No consumption pattern spec. What does the agent do with search results? When is memory lookup cheaper than rediscovery?
  • Agent behavior during live contradictions. No spec for what happens when agent encounters conflicting memories mid-task.
  • Cold-start and adoption friction. Who writes the first memories? How does an empty .memory/ become useful?
  • Eviction policy under quota. No memory ceiling defined. When it fills up, what gets deleted?
  • Why three memory types? decisions/corrections/patterns split is intuitive but never justified from user needs.
  • Correction scoring bias. +1.0 bonus is arbitrary. No measurement backing. Biases toward negative patterns.

Advisor Responses

The Contrarian 3 blockers found

Three critical failure modes: (1) Cache TTL creates write-read consistency bug at minute 50. (2) Contradiction keyword overlap >0.5 creates false positives constantly. (3) Graduation grepping at 60% is pattern-matching theatre.

Strong: frontmatter schema clean, pull-based retrieval right, separating structured memory from auto-memory architecturally sound.

Fix cache invalidation and graduation paths before shipping.

The First Principles Thinker trust problem

GC is a trust problem, not a mechanical problem. Contradiction detection produces false positives. Cache invalidation masks consistency bug. Race conditions for concurrent writes unaddressed.

Real risk: two separate systems (auto-memory + .memory/) without explaining why they can't be unified. Bifurcation doubles maintenance.

The Expansionist ship it

Pull-based progressive disclosure is exactly right. Frontmatter enforcement means every memory is queryable from day one. Composite routing keys scale from 10 to 10,000 memories. This unlocks agent learning loops — a 10x surface.

Push: if corrections timestamp why they superseded a decision, could weight that in future searches.

The Outsider operational gaps

Curse of knowledge. When agent gets search results, what does it actually do? Three gaps: no usage budget, correction handling broken during live execution, scoring function arbitrary (why 3.0 vs 2.0?).

Missing: test validation with 10 realistic memory files and a dozen queries.

The Executor reorder + gaps

Implementation order backwards — build GC first. Correction bonus biases negative patterns. No versioning or conflict resolution.

Missing: quota ceiling, audit trail, first-run bootstrap story.

Monday: build GC as pure functions with tests, then wire.
